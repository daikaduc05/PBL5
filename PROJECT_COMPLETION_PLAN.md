# PoseTrack Project Completion Plan

## 1. Purpose

This file is the working source of truth for the remaining work needed to call
the project complete.

When documents conflict, trust these files in this order:

1. `PROJECT_COMPLETION_PLAN.md`
2. `status.md`
3. `backend/backend_api_spec.md`

Do not use `OVERVIEW.md` as the source of truth. It is outdated and currently
does not match the real codebase state.

## 2. Current Truth

The project already has a real technical core:

- FastAPI backend is running with session, device, command, result, media, job,
  and history routes.
- Raspberry Pi agent can register, heartbeat, poll commands, and execute the
  replay/send flow.
- ZeroMQ worker can receive frames, run pose inference, and write result files.
- Result API can read processed sessions from
  `backend/workers/results/<session_key>`.
- Mobile app main MVP path is partially real:
  `Connect -> Capture -> Processing -> Result`.

The current real orchestration path is:

```text
Mobile App
-> POST /session/create
-> POST /devices/{device_id}/commands
Pi Agent
-> GET /devices/{device_id}/commands/pending
-> PATCH /devices/{device_id}/commands/{command_id}/status
Worker
-> writes backend/workers/results/<session_key>
Mobile App
-> GET /devices/{device_id}/commands/{command_id}
-> GET /results/{session_key}
```

## 3. Why The Project Is Not Complete Yet

The project is not complete because the system still has important gaps between
"real MVP demo" and "finished project":

- The canonical real path is `session + command + results`, but `history` still
  depends on the older `jobs` model.
- `/jobs` processing is still stubbed in `backend/app/services/job_service.py`.
- The Pi agent is still strongest in replay mode through `frames_dir`; live
  camera capture is not finished.
- `capture_photo` is not implemented in the Pi agent.
- `Connect` reads real backend data, but its actions still behave like refresh
  actions rather than true network pairing or scanning.
- `Home` is still mostly static demo content.
- `Settings` still use in-memory mock settings rather than persistent storage.
- Test coverage is very thin. There is no real end-to-end test proving the full
  phone -> backend -> Pi -> worker -> result loop on actual hardware.

## 4. Completion Goal

The project can be called complete only when all of the following are true:

- There is one canonical orchestration path for the user-facing flow.
- Every app-started capture creates records that can be traced from session,
  command, processing status, result, and history.
- The Pi supports real live capture modes, not only folder replay.
- The main mobile screens show live data instead of hardcoded demo data.
- Legacy mock or stub paths are either removed from the main flow or clearly
  isolated as demo-only fallbacks.
- Docs match the code.
- End-to-end verification has been run on the real stack.

## 5. Architecture Decision To Keep

The project should continue with this rule:

- `session + command + results` is the canonical orchestration path.

This means:

- Do not expand the fake `/jobs` thread as the main execution model.
- `history` must be aligned to real session runs produced by the canonical flow.
- If `jobs` are kept, they must become a thin real execution record tied to the
  canonical path, not a separate fake pipeline.

## 6. Ordered Work Plan

### Phase 1: Unify The Record Model

Goal:
Make session, command, result, and history describe the same real capture run.

Tasks:

- Refactor backend history so it is driven by real app-started runs, not by the
  old stub-only job path.
- Ensure every app-started capture has a traceable record containing at least:
  `session_id`, `session_key`, `device_id`, `command_id`, capture mode, start
  time, completion state, and result summary.
- Decide whether to:
  - retire `JobModel` from the user-facing flow, or
  - turn it into a real execution record generated by the canonical flow.
- Update mobile history screens to consume the unified backend model.

Priority files:

- `backend/app/api/routes/history.py`
- `backend/app/services/job_service.py`
- `backend/app/api/routes/job.py`
- `mobile_app/lib/services/api_service.dart`
- `mobile_app/lib/screens/history_screen.dart`

Definition of done:

- A capture started from the current app flow appears in `History` without any
  manual or fake `/jobs` step.
- `History` and `Result` both point to the same real session run.
- No user-facing screen depends on stub job progress to look correct.

### Phase 2: Finish Real Pi Capture Modes

Goal:
Move from replay-first behavior to true Raspberry Pi capture support.

Tasks:

- Implement live video capture on the Pi agent.
- Implement `capture_photo` on the Pi agent.
- Keep folder replay only as an explicit demo/testing fallback.
- Confirm command payloads support both replay mode and real camera mode.
- Make `stop_recording` meaningful for the real capture path.

Priority files:

- `backend/pi_agent/pi_agent.py`
- `backend/pi_agent/pi_zmq_sender.py`
- any Pi camera helper module introduced for real capture
- `mobile_app/lib/screens/capture_control_screen.dart`

Definition of done:

- The Pi can execute a real app-triggered image capture.
- The Pi can execute a real app-triggered short video capture.
- Replay mode still works, but it is no longer the only practical path.

### Phase 3: Make Mobile Screens Truthful

Goal:
Remove the remaining "looks complete but still demo-like" gaps in the app.

Tasks:

- `Home`: replace hardcoded service cards and recent session summary with live
  backend data.
- `Connect`: either implement true scan/pair semantics or relabel the actions
  so they accurately describe refresh/status behavior.
- `Settings`: persist server and Pi configuration across app restarts.
- Reduce legacy fallback logic on screens that now have backend ids available.
- Make failure states clearer when backend, Pi, or result session is missing.

Priority files:

- `mobile_app/lib/screens/home_screen.dart`
- `mobile_app/lib/screens/device_connection_screen.dart`
- `mobile_app/lib/screens/settings_screen.dart`
- `mobile_app/lib/services/mock_device_connection_service.dart`
- `mobile_app/lib/services/mock_pose_tracking_service.dart`

Definition of done:

- The app home screen reflects real runtime status.
- Settings survive app restart.
- Screen labels and actions match their real behavior.

### Phase 4: Close The Legacy Mock Gap

Goal:
Clearly separate production flow from legacy demo fallback.

Tasks:

- Audit remaining `MockPoseTrackingService` usage.
- Remove legacy result/finalize paths from the main flow where real backend ids
  are required.
- Keep demo-only behavior only if it is explicitly labeled and isolated.
- Remove dead compatibility branches that no longer serve the current app path.

Priority files:

- `mobile_app/lib/services/mock_pose_tracking_service.dart`
- `mobile_app/lib/screens/processing_status_screen.dart`
- `mobile_app/lib/screens/result_screen.dart`
- `mobile_app/lib/navigation/app_routes.dart`

Definition of done:

- The default user path is fully backend-driven.
- Any remaining demo fallback is clearly optional and not confused with the
  real project path.

### Phase 5: Verification And Documentation

Goal:
Prove the project works as a system and leave clean docs for continuation.

Tasks:

- Add backend tests for command lifecycle, history/result consistency, and
  result service behavior.
- Add Flutter tests for the real navigation and API-backed screen states.
- Run a manual end-to-end test on the real stack:
  app -> backend -> Pi -> worker -> result -> history.
- Document the exact setup steps for backend, worker, Pi agent, and mobile app.
- Remove or rewrite stale docs that still describe the old architecture.

Priority files:

- `backend/backend_api_spec.md`
- `status.md`
- `OVERVIEW.md`
- test files to be added in backend and mobile app

Definition of done:

- There is one current setup guide and one current completion roadmap.
- The team can reproduce the full flow without guessing hidden steps.

## 7. Immediate Next Task

If work resumes from this snapshot, start here:

1. Phase 1: unify the record model and fix the `history` / `jobs` mismatch.
2. Phase 2: implement real Pi live capture and `capture_photo`.
3. Phase 3: make `Home`, `Connect`, and `Settings` fully truthful.
4. Phase 5: run and document the first full end-to-end verification.

Do not start with new UI polish. The biggest blockers are still data model
alignment and real device capture.
